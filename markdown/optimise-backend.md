# バックエンド（Go）パフォーマンス最適化調査

## 概要

UX Music の Go バックエンド全体を対象に、メモリ消費・CPU 消費・サブプロセス効率の観点からコードを調査した結果をまとめる。
フロントエンド側の最適化計画は `optimise.md` を参照。

---

## 影響度の分類

| レベル | 意味 |
|--------|------|
| **高** | 再生中に常時走る処理、または大規模ライブラリで O(n) 以上の悪化が見込まれる |
| **中** | 特定操作時に発生するが、体感に影響しうる |
| **低** | 改善余地はあるが体感への影響は限定的 |

---

## 1. オーディオ再生ホットパス（影響度: 高）

### 対象ファイル
- `pkg/audio/player.go`

### 1-1. processAudio コールバック内での FFT バッファ確保

**場所**: `processAudio` 関数内、FFT サンプル送信部分

オーディオコールバックは毎秒数十回呼び出される。呼び出しのたびに `make([]float64, ...)` で新規スライスを確保しており、GC 圧力の原因となる。

**改善案**: `Player` 構造体にあらかじめ FFT 用バッファを確保し、コールバックでは再利用する。

### 1-2. リングバッファへの書き込みで毎サンプル modulo 演算

**場所**: `decoderLoop` 関数内、サンプル変換ループ

`(writePos + int64(i)) % int64(p.ringBufSize)` が全サンプルに対して実行されている。44.1kHz ステレオの場合、毎秒約 88,200 回の剰余演算が走る。

**改善案**: バッチ単位で書き込み先のオフセットを計算し、`copy()` でまとめて転送する。リングバッファサイズを 2 の冪に揃えればビットマスクで代用可能。

### 1-3. FFT プロセッサのバッファ蓄積

**場所**: `initFFT` 関数、FFT 処理ゴルーチン

FFT 結果チャネル（バッファサイズ 4）に詰まったサンプルが蓄積する可能性がある。FFT 処理が遅延した場合、古いデータが溜まり続ける。

**改善案**: チャネルの代わりにアトミックなダブルバッファ（最新結果のみ保持）に切り替える。古いデータは自動的に破棄される。

## 2. ライブラリスキャン（影響度: 高）

### 対象ファイル
- `app_scanner.go`
- `internal/scanner/scanner.go`
- `internal/scanner/artwork.go`

### 2-1. 無制限のゴルーチン生成

**場所**: `internal/scanner/scanner.go` の `walkDirectory` 関数

オーディオファイル 1 件ごとに 1 ゴルーチンを生成している。10,000 曲規模のライブラリでは同時に 10,000 ゴルーチンが起動し、ファイル記述子やメモリを大量消費する。

**改善案**: `runtime.NumCPU()` 程度のワーカープールを導入し、セマフォ（バッファ付きチャネル）でゴルーチン数を制限する。

### 2-2. 毎回全ライブラリを再スキャン

**場所**: `app_scanner.go` の `ScanLibrary` 関数

スキャンのたびにライブラリ全体をロードし、全パスのインデックスを再構築している。差分検出（ファイルの mtime 比較）がないため、変更のないファイルも毎回パースされる。

**改善案**: 前回スキャン時のファイルパス + mtime をキャッシュし、変更があったファイルのみ再パースする。

### 2-3. アートワーク抽出の重複

**場所**: `internal/scanner/artwork.go` の `extractAndSaveArtwork`

全曲に対して個別にアートワーク抽出（FFmpeg サブプロセス）を実行している。同一アルバムの曲はアートワークが同じであることが多い。

**改善案**: アルバム名（またはアルバム + アーティスト）をキーにしてアートワーク抽出結果をキャッシュし、同一キーの2曲目以降はスキップする。

---

## 3. JSON ストア（影響度: 中）

### 対象ファイル
- `internal/store/store.go`

### 3-1. 保存時に毎回 JSON 全体を再マーシャル + 全書き込み

**場所**: `Save` メソッド

`json.MarshalIndent()` で整形付き JSON を生成し、ファイル全体を上書きしている。大規模ライブラリ（10,000 曲以上）では数 MB 規模の書き込みが発生する。

**改善案**:
- 本番用途ではインデント無し（`json.Marshal()`）に切り替え、出力サイズと CPU 時間を削減する
- ダーティフラグを導入し、変更がない場合は書き込みをスキップする
- 将来的にはバイナリフォーマット（gob, msgpack 等）への移行も検討

### 3-2. Load の都度フルデシリアライズ

**場所**: `Load` メソッド

呼び出しのたびに `os.ReadFile` + `json.Unmarshal` が走る。頻繁に参照されるデータ（ライブラリ、ラウドネス値）にはインメモリキャッシュがない。

**改善案**: ロード結果をメモリにキャッシュし、`Save()` 時にキャッシュを無効化するだけにする。

---

## 4. ラウドネス解析 / ノーマライズ（影響度: 中）

### 対象ファイル
- `app_normalize.go`
- `pkg/normalize/`

### 4-1. ファイルごとに FFmpeg サブプロセスを個別起動

**場所**: `NormalizeStartJob` 内のクロージャ、`queueLoudnessAnalysis` 内のクロージャ

1 曲につき 1 回の FFmpeg プロセス起動がある。プロセス生成のオーバーヘッド（fork + exec）が曲数に比例して増大する。

**改善案**:
- 複数ファイルを 1 回の FFmpeg 呼び出しでバッチ処理する（`filter_complex` で並列解析）
- 解析済みファイルの結果をキャッシュし、同じファイルの重複解析を防止する

### 4-2. ワーカー数の上限

現在のワーカー数は `max(1, GOMAXPROCS-1)` で上限 4。Apple Silicon Mac では 8〜12 コアあるため、I/O バウンドな FFmpeg 処理にはもう少し並列度を上げられる可能性がある。

**改善案**: ベンチマークで最適なワーカー数を計測し、設定可能にする。

---

## 5. FLAC インデックス生成（影響度: 中）

### 対象ファイル
- `app_scanner.go` の `BuildFLACIndexes`

### 5-1. ジョブチャネルのバッファサイズ

ジョブチャネルが `total`（全 FLAC ファイル数）でバッファされている。数千ファイルの場合、チャネル確保だけでメモリを消費する。

**改善案**: チャネルバッファを小さい固定サイズ（16〜64 程度）にし、送信側をストリーミングに変更する。

### 5-2. 変更のないファイルもインデックス再生成

既にキャッシュが存在するファイルのスキップ処理はあるが、ファイルの更新日時チェックがない可能性がある。ファイルが変更されていなければインデックス再生成は不要。

**改善案**: キャッシュファイルの mtime と元ファイルの mtime を比較し、更新があった場合のみ再生成する。

---

## 6. Wear サーバ（影響度: 低）

### 対象ファイル
- `app_wear.go`

### 6-1. リクエストごとに pathToIDMap を再構築

**場所**: `wearLoudnessHandler`

リクエストのたびにライブラリ全体を走査して `pathToIDMap` を構築している。曲数 × リクエスト数の計算量になる。

**改善案**: マップをキャッシュし、ライブラリ更新時のみ再構築する。

### 6-2. /wear/songs でページネーション未対応

**場所**: `wearSongsHandler`

全曲のメタデータを一括で JSON レスポンスとして返している。大規模ライブラリでは応答サイズが巨大になる。

**改善案**: ページネーション（offset/limit パラメータ）を導入し、クライアントが必要な分だけ取得できるようにする。

### 6-3. トランスコードのキャッシュ管理

**場所**: `getOrTranscode`

トランスコード結果はキャッシュされるが、アプリクラッシュ時の一時ファイルが残る可能性がある。

**改善案**: 起動時に孤立した一時ファイルをクリーンアップする処理を追加する。

---

## 7. 歌詞同期（影響度: 低〜中）

### 対象ファイル
- `internal/lyricssync/syncer.go`
- `internal/lyricssync/whisper_runner.go`

### 7-1. 逐次的なサブプロセス実行

**場所**: `syncer.go` の同期処理

vocal-ml、vocal-focus、plain の 3 ソースに対して逐次的にサブプロセスを実行している。独立した処理は並列化できる。

**改善案**: vocal-ml と vocal-focus の抽出を並列実行し、完了を待ち合わせる。

### 7-2. コマンドパス解決の繰り返し

**場所**: `whisper_runner.go` の環境変数参照

Whisper CLI のパス解決が呼び出しのたびに行われている。

**改善案**: `sync.Once` で初回のみ解決し、以降はキャッシュを使う（一部は既に `sync.Map` でキャッシュ済みの可能性あり。要確認）。

## 8. GPU 利用について

現時点で GPU を直接利用するコードはない。以下の処理は GPU オフロードの候補となりうる。

| 処理 | 現状 | GPU オフロードの可能性 |
|------|------|----------------------|
| FFT（周波数解析） | CPU（go-dsp/fft） | Metal Compute Shader で高速化可能だが、2048 点 FFT では効果限定的 |
| EQ フィルタ | CPU（biquad フィルタ） | リアルタイムオーディオには不向き（レイテンシ増加） |
| Demucs（ボーカル分離） | サブプロセス経由 | 既に `UXMUSIC_LYRICS_SYNC_DEMUCS_DEVICE` で mps/cuda 指定可能 |
| アートワーク変換 | FFmpeg サブプロセス | VideoToolbox 経由で GPU エンコードは可能だがオーバーヘッド大 |

**結論**: 現在の処理規模では GPU オフロードの効果は限定的。Demucs の GPU 利用設定が既にあるため、追加の GPU 最適化の優先度は低い。

## 優先着手順の提案

| 順位 | 項目 | 影響度 | 実装難度 |
|------|------|--------|----------|
| 1 | スキャナのゴルーチン制限（2-1） | 高 | 低 |
| 2 | FFT バッファの事前確保（1-1） | 高 | 低 |
| 3 | リングバッファ書き込みのバッチ化（1-2） | 高 | 中 |
| 4 | Store のインメモリキャッシュ（3-2） | 中 | 低 |
| 5 | アートワーク抽出の重複排除（2-3） | 中 | 低 |
| 6 | ラウドネス解析のバッチ化（4-1） | 中 | 中 |
| 7 | 差分スキャン（2-2） | 高 | 高 |
| 8 | JSON フォーマット簡素化（3-1） | 中 | 低 |
| 9 | Wear サーバのキャッシュ改善（6-1, 6-2） | 低 | 低 |
| 10 | 歌詞同期の並列化（7-1） | 低 | 中 |

---

## メモ

- 最適化は `optimise.md`（フロントエンド）と合わせて「計測 -> 仮説 -> 変更 -> 再計測」で進める
- `app_perf.go` に既存のプロファイリング機能があるため、これを活用して計測基盤とする
- 各項目は独立して着手可能。Phase 分けはフロントエンド側と同期する必要はない

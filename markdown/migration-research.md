# Electron から Wails への移行難易度調査レポート (ハイブリッド MTP 戦略反映版)

## 1. 総合評価
**難易度：低 (Low) 〜 中 (Medium)**
*(以前の「中〜高」からさらに下方修正)*

**理由**: 
MTP（楽曲転送）機能を Go へ移植せず、**「必要な時だけ起動する Node.js サブプロセス」** として維持するハイブリッド戦略を採用することで、懸念されていた「MTP プロトコルスタックの再実装」という最大のリスクが解消されました。

## 2. 移行戦略の変更とメリット

### A. MTP ハイブリッド・ワーカー戦略 (New Implementation Plan)
- **方針**: 現在の `Kalam.js` (JavaScript) をそのまま活用し、Wails (Go) から必要な時だけ Node.js プロセスを起動して通信する。
- **メリット**: 
  - **開発コストの劇的な削減**: 数千行の通信プロトコル移植が不要。
  - **安定性**: 既に動作実績のある `Kalam.js` をそのまま利用。
  - **リソース効率**: ユーザー提案の通り、転送時以外は Node.js プロセスを終了させておくことで、常用メモリを数MB単位に抑える Wails の利点を損なわない。

### B. 現状の進捗状況
1. **バックエンドロジック (完了)**: プレイリスト、歌詞、解析等の主要ロジックは Go 移行済み。
2. **通信基盤 (完了)**: JSON ベースの双方向通信プロトコルは Sidecar 実装で確立済み。Wails でも同様の構造で MTP ワーカーと通信可能。

## 3. 残存作業と難易度

| 項目 | 難易度 | 内容 |
| :--- | :--- | :--- |
| **UI 移植** | 低 | HTML/CSS はそのまま流用。IPC 呼び出しを Wails 形式に置換。 |
| **MTP 連携** | 中 | Go から Node.js を起動し、標準入出力で `Kalam.js` とやり取りするブリッジの実装。 |
| **パッケージング** | 中 | Wails バイナリに Node.js ランタイムまたはスクリプトを同梱する設定。 |

## 4. 推奨されるフェーズ分け (最終案)

1. **Phase 1: ロジック分離 (完了)**
   - **Status**: 完了 (Go Sidecar への移行済み)

2. **Phase 2: MTP ブリッジの実装 (Next)**
   - 現在の Electron 環境下にて、MTP 処理を独立したスクリプトとして分離し、Go サイドカーから直接呼び出せるか検証する。

3. **Phase 3: Wails への完全統合**
   - Electron を廃止し、Wails のウィンドウに切り替える。

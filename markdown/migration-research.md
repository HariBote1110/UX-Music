# Electron から Wails への移行難易度調査レポート (改訂版)

## 1. 総合評価
**難易度：高 (High) 〜 極めて高 (Critical)**

今回の IPC 不具合修正（`load-library` vs `library-loaded` の不一致等）を通じて、本アプリのフロントエンドとバックエンドが IPC イベント名を介して非常に密に、かつ「緩く」結合していることが浮き彫りになりました。Wails への移行は単なるフレームワークの載せ替えではなく、**「通信プロトコルとバックエンドロジックの完全な再定義」**を意味します。

## 2. 主要な技術的課題と障壁

### A. 通信定義の二重管理と型安全性の欠如
- **現状**: メインとレンダラーでイベント名が文字列で定義されており、今回のように定義が乖離するリスクが常にあります。
- **Wails 移行の障壁**: Wails では Go の関数が直接 JS にバインドされるため、まずバックエンドの全機能を Go のメソッドとして再定義する必要があります。

### B. MTP プロトコルスタック (Kalam.js) の移植
- **現状**: [Kalam.js](file:///Users/yuki/GitHub/UX-Music/src/main/mtp/Kalam.js) は Pure JavaScript による MTP 実装です。
- **障壁**: Go にはこれと同等に柔軟な Pure Go MTP ライブラリは少なく、cgo を用いて `libmtp` 等の C ライブラリと連携させるか、Kalam.js を Go で書き直す（高コスト）必要があります。これが移行の最大の「詰め」になります。

### C. ネイティブ依存モジュールの代替選定
Go への移行にあたり、以下の npm パッケージを Go パッケージへ置換する必要があります：

| カテゴリ | 現状 (npm) | Go 代替案 | 備考 |
| :--- | :--- | :--- | :--- |
| USB検出 | `usb-detection` | `google/gousb` | libusb への依存が発生。 |
| 画像処理 | `sharp` | `h2non/bimg` | libvips 依存。ビルドが複雑化する可能性。 |
| メタデータ | `music-metadata` | `dhowden/tag` | 標準的なタグ読み書きは対応可能。 |
| 楽曲解析 | `music-tempo`, `wav-decoder` | 自作または ffmpeg 依存 | Go での信号処理実装が必要。 |

### D. 外部バイナリ (FFmpeg) の統合
- `ffmpeg-static` 等の恩恵が受けられなくなるため、Wails のサイドカーバイナリ機能を使用して FFmpeg を同梱し、Go の `os/exec` で制御する設計への変更が必要です。

## 3. 移行のメリットと期待される改善
- **パフォーマンス**: BPM解析やラウドネス解析を Goroutine で並列実行することで、Node.js の Worker 処理よりも効率的かつ高速に処理できる可能性があります。
- **メモリ効率**: Electron の数百MBというフットプリントを、数MB〜数十MB単位まで削減可能です。

## 4. 推奨されるフェーズ分け

1. **Phase 1: 通信層の抽象化 (JS側)**
   - `ipc.js` をさらに抽象化し、ビジネスロジックから `electronAPI` への直接依存を排除する。
2. **Phase 2: MTP 検証 (Go側)**
   - Go から Walkman へのファイル転送が安定して行えるライブラリ構成を確定させる。
3. **Phase 3: 段階的ビルド**
   - まずは音楽プレイヤー機能のみを Wails で実装し、MTP や解析機能は徐々に追加する。
